## Objective
To quantify the performance impact of Transparent Huge Pages (THP) on Redis, specifically focusing on hardware-level metrics (TLB hits/misses), memory allocation behavior, and tail latency (P99) under system stress.

## Key Technical Decisions
1.  **Strict Isolation**: Pinned the Redis server to Core 0 and the benchmark client to Core 1 to eliminate context switching and cache interference.
2.  **Noise Generation**: Used `stress-ng` on remaining cores (2-5) to simulate a realistic production environment and prevent background task migration to test cores.
3.  **Hardware Monitoring**: Leveraged `perf stat` to capture dTLB and iTLB loads/misses directly from the CPU performance counters.
4.  **State Management**: Implemented a "Pre-flight Capture" and "Cleanup" logic to ensure system settings (THP, Memory Overcommit) are restored after the benchmark.
5.  **Memory Optimization**: Set `overcommit_memory=1` to ensure stable `BGSAVE` (Copy-on-Write) operations during the test.

## How to Analyze Results
- **TLB Hit Rate**: Calculated as `(Loads - Misses) / Loads`. THP should significantly increase this rate.
- **THP Fallback**: If `thp_fault_fallback` is high, it indicates memory fragmentation is preventing THP from being utilized, negating its benefits.
- **P99 Latency**: Compare the P99 across `always`, `madvise`, and `never` modes to identify the tail-latency trade-offs mentioned in the research paper.
